## Conceitos Fundamentais

O Domínio 6 do CISSP cobre metodologias e técnicas para avaliar a eficácia dos controles de segurança e identificar vulnerabilidades antes que atacantes o façam. Representa 12% do peso do exame e é fundamental para provar que controles funcionam conforme esperado.

**Tipos de Avaliações de Segurança**: Vulnerability Assessment identifica e classifica vulnerabilidades conhecidas sem tentar explorá-las — usa scanners automatizados (Nessus, Qualys, OpenVAS). Penetration Test simula um atacante real tentando explorar vulnerabilidades para alcançar um objetivo definido. Red Team simula um adversário persistente com TTPs realistas ao longo de semanas/meses. Bug Bounty utiliza pesquisadores externos de segurança para identificar vulnerabilidades em sistemas definidos.

**Conhecimento do Alvo**: Black-box (caixa preta) — o testador tem o mínimo de informações, simulando um atacante externo sem conhecimento prévio; White-box (caixa branca) — o testador tem acesso completo a documentação, código-fonte e arquitetura; Gray-box (caixa cinza) — conhecimento parcial, simulando um insider ou atacante com informações limitadas.

**Tipos de Pentest por Escopo**: Network Penetration Test — foca em infraestrutura de rede, sistemas e serviços expostos; Web Application Pentest — foca em aplicações web seguindo OWASP Testing Guide; Social Engineering — phishing, vishing, pretexting; Physical Pentest — testa controles de acesso físico; Mobile Application Pentest; Cloud Infrastructure Pentest.

**Auditoria de Segurança**: Avaliação formal e documentada da conformidade de controles com políticas, normas e requisitos regulatórios. Pode ser interna (equipe de auditoria interna) ou externa (auditores independentes). SOC 2 Type II e ISO 27001 são exemplos de auditorias com escopo e critérios definidos por padrões reconhecidos.

**Code Review e SAST/DAST**: Static Application Security Testing (SAST) analisa código-fonte ou bytecode sem executar o programa para identificar vulnerabilidades (SQL injection, XSS, buffer overflow, hardcoded credentials). Dynamic Application Security Testing (DAST) testa a aplicação em execução simulando requisições maliciosas. IAST (Interactive) combina SAST e DAST via instrumentação em runtime. RASP (Runtime Application Self-Protection) protege aplicações em produção.

**Gestão de Vulnerabilidades**: Processo contínuo de identificação, classificação, priorização, remediação e verificação de vulnerabilidades. CVSS (Common Vulnerability Scoring System) fornece pontuação padronizada de 0 a 10 para priorização. SLA de remediação por criticidade: Crítico ≤ 15 dias, Alto ≤ 30 dias, Médio ≤ 90 dias, Baixo ≤ 180 dias.

## Frameworks de Referência

- **OWASP Testing Guide (OTG)**: Guia abrangente para teste de segurança de aplicações web. Cobre 11 categorias de teste: Information Gathering, Configuration Management, Authentication, Session Management, Authorization, Data Validation, Error Handling, Cryptography, Business Logic, Client Side, API Testing. O OWASP Web Security Testing Guide v4.2 é o documento mais completo para metodologia de teste web. Complementado pelo OWASP Top 10 para priorização de vulnerabilidades.

- **NIST SP 800-115**: Technical Guide to Information Security Testing and Assessment define tipos de técnicas de teste (revisão, identificação alvo, análise de alvo, validação de alvo), considera como cada técnica apoia os objetivos de segurança e fornece diretrizes para planejamento, execução e geração de relatórios. Define o processo de teste em quatro fases: planejamento, descoberta, ataque e relatório.

- **PTES (Penetration Testing Execution Standard)**: Define sete fases para um pentest completo: Pre-engagement Interactions (escopo, regras de engajamento, legalidades), Intelligence Gathering (OSINT, reconhecimento), Threat Modeling, Vulnerability Analysis, Exploitation, Post-Exploitation, e Reporting. PTES é a referência mais completa para metodologia de pentest profissional.

## Controles Principais

1. **Programa de Gestão de Vulnerabilidades**: Implementar ciclo contínuo de discovery (scan automatizado semanal para ativos críticos, mensal para demais), análise e priorização por CVSS + contexto de negócio, atribuição de proprietário de remediação, acompanhamento com SLAs definidos e verificação pós-correção. Usar plataforma centralizada (Tenable, Qualys, Rapid7) com dashboards por equipe.

2. **Testes de Penetração Periódicos**: Realizar pentest externo anual por firma independente especializada. Pentest de aplicações críticas a cada release major ou anualmente. Exercícios de Red Team a cada 18-24 meses para avaliar capacidade de detecção e resposta. Garantir que todos os testes tenham escopo formal, regras de engajamento e relatório detalhado com evidências.

3. **Integração de SAST e DAST no Pipeline de CI/CD**: Integrar ferramentas de análise estática (SonarQube, Checkmarx, Semgrep) no pipeline de build como quality gates. Implementar DAST (OWASP ZAP, Burp Suite Enterprise) em ambientes de staging. Definir critérios de bloqueio de deploy para vulnerabilidades críticas e altas não remediadas.

4. **Revisões de Código de Segurança (Secure Code Review)**: Implementar processo de peer review com foco em segurança para todos os commits em código de sistemas críticos. Treinar desenvolvedores em secure coding (OWASP Top 10, CWE/SANS Top 25). Usar threat modeling para guiar o foco das revisões de código em áreas de maior risco.

5. **Controle de Acesso ao Ambiente de Teste**: Garantir que ambientes de teste e desenvolvimento não contenham dados de produção reais sem anonimização/mascaramento. Controlar credenciais de teste e ambientes de pentest para evitar uso indevido. Revogar credenciais temporárias criadas durante testes imediatamente após conclusão.

6. **Log e Auditoria de Resultados de Testes**: Manter registro histórico de todos os assessments realizados, vulnerabilidades identificadas, status de remediação e métricas de tendência. Apresentar evolução da postura de segurança ao longo do tempo à liderança. Usar dados históricos para benchmarking e justificativa de investimentos.

7. **Testes de Controles de Segurança (Control Testing)**: Além de testes de penetração, testar periodicamente se controles funcionam conforme configurado: simular ataques de phishing para testar detecção por e-mail gateway e comportamento de usuários, simular malware para testar EDR, simular lateral movement para testar detecção de rede. Usar Purple Team para calibrar controles contra TTPs reais.

## KPIs do Domínio

| Métrica | Fórmula | Benchmark |
|---------|---------|-----------|
| SLA de Remediação — Crítico | % de vulnerabilidades críticas remediadas dentro de 15 dias | ≥ 95% |
| SLA de Remediação — Alto | % de vulnerabilidades altas remediadas dentro de 30 dias | ≥ 90% |
| Vulnerabilidades Críticas Abertas | Contagem de CVEs críticos (CVSS ≥ 9.0) não remediados | 0 após SLA |
| Cobertura de Scan de Vulnerabilidade | (Ativos críticos escaneados no período / Total de ativos críticos) × 100 | 100% semanal |
| Findings Críticos Reabertos | % de vulnerabilidades críticas que reapareceram após remediação | ≤ 5% |

## Aplicação por Situação

- **S21 – Vulnerabilidades**: Ao receber um relatório de scan com volume elevado de vulnerabilidades, priorizar pela fórmula: CVSS Base Score + contexto de negócio (sistema exposto externamente, dados sensíveis, criticidade do sistema) + exploitabilidade (CVE com exploit público ou ativo em wild). Criar plano de remediação em três ondas: Onda 1 (0-30 dias) — todos os críticos e altos em sistemas expostos externamente; Onda 2 (30-90 dias) — todos os altos em sistemas internos e médios em sistemas expostos; Onda 3 (90-180 dias) — médios e baixos restantes. Monitorar progresso semanalmente e reportar métricas ao negócio mensalmente. Para vulnerabilidades que não podem ser remediadas imediatamente, implementar compensating controls documentados.
